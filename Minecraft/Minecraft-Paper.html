Paper配置解析

    Paper是PaperMC团队基于Spigot制作的一个更高性能和更多BUG修复，同时还加入了对Velocity(同样由PaperMC开发)服务端的支持，Paper最早的名称为PaperSpigot，后被更名为Paper

Paper在其Bukkit API和Spigot API的基础上同样添加了很多自己的API接口（PaperAPI）并且加入了大量的异步化线程支持，PaperMC还改进并优化了很多Spigot中存在的缺陷，显著提高了性能

相较于Spigot，Paper中的Timings V2中包含了更多的信息（注：后续PaperMC官方会将内置的Timings逐步替换为Spark）

Paper改进了很多Bukkit与Spigot的API，并保留了对原有的Bukkit和Spigot插件的支持，Paper拥有性能更高的Anti X-ray反矿透系统，相比较插件而言Paer自带的功能拥有更高的性能更低的资源占用

 

本片文章部分内容参考自：PaperWiki，MineWiki，我的世界服务器搭建指南，Minecraft插件百科

首先我们先做一下基础准备工作，生成配置文件

  1. 直接去PaperMC官网下载服务端，相较于Spigot无需其他操作直接下载
  2. 编写启动脚本并启动服务端

设置区块的读写操作线程数

io-threads: -1

 以下为具体参数：

  • 整数型

增加这个值的前提是你的硬盘写入和读取性能都很棒（至少是PCIE3.0的标准），设置低于0以下的值（如 -1 等）将会使用一个线程

 

设置并发区块生成的线程

worker-threads: -1

 以下为具体参数：

  • 整数型

设置低于0以下的值（如 -1 等），服务端将根据可用CPU核心的数量来确定最佳值，这会限制在可用核心的四分之一，对于核心少的服务器来说可能更少，注意：不计算超线程

 以下为具体参数：

是否允许玩家相互碰撞，这个选项可以被与记分板交互的插件破坏，如果在使用此选项时遇到问题，请尝试不安装插件启动服务端

 

是否发送完整的碰撞数据

send-full-pos-for-hard-colliding-entities: true

 以下为具体参数：

  • false - 否
  • true - 是

玩家与船只和矿车的碰撞通常会引起客户端/服务器的分歧，这可能会导致玩家出现错误的行为，此设置会通过发送涉及冲突的实体的精确位置来缓解这种问题，启用此功能将占用更多网络带宽

 以下为具体参数：

  • chat-executor-max-size: -1·································聊天线程池中允许的最大线程数，默认值为-1，表示没有最大值

设置数据压缩处理等级

compression-level: default

 以下为具体参数：

  • -1··········································································关闭压缩
  • 1 - 10····································································压缩等级1到10，默认为6

当群组端和子服处于一个网络下建议关闭，因为浪费性能且没什么卵用，数值越大（-1除外）需要耗费的CPU性能越多，同时越节省服务器带宽，反之亦然（看你的服务器性能和带宽适当调整）

  • 这个和原版服务端配置文件 server.properties 中的 network-compression-threshold 有些许的不同，那个是控制数据包的大小，这个是数据包的压缩等级
  • 注意：Paper目前正在使用 Velocity 的压缩器，它默认的压缩级别为 6 作为性能和数据传输之间的平衡点

 

是否修复 MC-4 的问题

fix-entity-position-desync: true

 以下为具体参数：

  • false - 关闭
  • true - 开启

这玩意属于是千年老痰了，大概意思是当物品落在方块的边缘时，客户端有时会让它掉到其他地方，而服务器则将其留在原地，这是因为客户端的一种预测，但是很显然这是错误的预测

 

是否允许插件在加载之前先加载permissions.yml

 以下为具体参数：

在插件之前加载 bukkit 的 permission.yml 文件，允许他们在启动时立即检查文件内容，如果你的 permission.yml 没有任何内容可以关闭，不过这没什么卵用，因为先加载和后加载对性能影响不大
